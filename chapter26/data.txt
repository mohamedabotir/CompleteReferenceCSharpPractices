	<h2 id="remarks">Remarks</h2>
	<p>The <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> class instance acts as a session to send HTTP requests. An <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> instance is a collection of settings applied to all requests executed by that instance. In addition, every <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> instance uses its own connection pool, isolating its requests from requests executed by other <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> instances.</p>
<p>Derived classes should not override the virtual methods on the class. Instead, use a constructor overload that accepts <a href="system.net.http.httpmessagehandler?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpMessageHandler</a> to configure any pre- or post-request processing.</p>
<p>By default on .NET Framework and Mono, <a href="system.net.httpwebrequest?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpWebRequest</a> is used to send requests to the server. This behavior can be modified by specifying a different channel in one of the constructor overloads taking a <a href="system.net.http.httpmessagehandler?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpMessageHandler</a> instance as parameter. If features like authentication or caching are required, <a href="system.net.http.webrequesthandler?view=net-6.0" class="no-loc" data-linktype="relative-path">WebRequestHandler</a> can be used to configure settings and the instance can be passed to the constructor. The returned handler can be passed to one of the constructor overloads taking a <a href="system.net.http.httpmessagehandler?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpMessageHandler</a> parameter.</p>
<p>If an app using <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> and related classes in the <a href="system.net.http?view=net-6.0" class="no-loc" data-linktype="relative-path">System.Net.Http</a> namespace intends to download large amounts of data (50 megabytes or more), then the app should stream those downloads and not use the default buffering. If the default buffering is used the client memory usage will get very large, potentially resulting in substantially reduced performance.</p>
<p>Properties of <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> should not be modified while there are outstanding requests, because it is not thread-safe.</p>
<p>The following methods are thread safe:</p>
<ol>
<li><p><a href="system.net.http.httpclient.cancelpendingrequests?view=net-6.0" class="no-loc" data-linktype="relative-path">CancelPendingRequests</a></p>
</li>
<li><p><a href="system.net.http.httpclient.deleteasync?view=net-6.0" class="no-loc" data-linktype="relative-path">DeleteAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.getasync?view=net-6.0" class="no-loc" data-linktype="relative-path">GetAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.getbytearrayasync?view=net-6.0" class="no-loc" data-linktype="relative-path">GetByteArrayAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.getstreamasync?view=net-6.0" class="no-loc" data-linktype="relative-path">GetStreamAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.getstringasync?view=net-6.0" class="no-loc" data-linktype="relative-path">GetStringAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.postasync?view=net-6.0" class="no-loc" data-linktype="relative-path">PostAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.putasync?view=net-6.0" class="no-loc" data-linktype="relative-path">PutAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.sendasync?view=net-6.0" class="no-loc" data-linktype="relative-path">SendAsync</a></p>
</li>
<li><p><a href="system.net.http.httpclient.patchasync?view=net-6.0" class="no-loc" data-linktype="relative-path">PatchAsync</a></p>
</li>
</ol>
<p><a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> is intended to be instantiated once and re-used throughout the life of an application. Instantiating an HttpClient class for every request will exhaust the number of sockets available under heavy loads. This will result in SocketException errors. Below is an example using HttpClient correctly.</p>
<pre><code class="lang-csharp">public class GoodController : ApiController
{
    private static readonly HttpClient HttpClient;

    static GoodController()
    {
        HttpClient = new HttpClient();
    }
}
</code></pre>
<pre><code class="lang-vb">  Public Class GoodController
    Inherits ApiController  

    Private Shared ReadOnly HttpClient As HttpClient  

    Shared Sub New()  
        HttpClient = New HttpClient()
    End Sub
End Class
</code></pre>
<p>The <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a> is a high-level API that wraps the lower-level functionality available on each platform where it runs.</p>

<h3 id="httpclient-and-net-core">HttpClient and .NET Core</h3>
<p>Starting with .NET Core 2.1, the <a href="system.net.http.socketshttphandler?view=net-6.0" class="no-loc" data-linktype="relative-path">System.Net.Http.SocketsHttpHandler</a> class instead of <code>HttpClientHandler</code> provides the implementation used by higher-level HTTP networking classes such as <code>HttpClient</code>. The use of <a href="system.net.http.socketshttphandler?view=net-6.0" class="no-loc" data-linktype="relative-path">SocketsHttpHandler</a> offers a number of advantages:</p>
<ul>
<li>A significant performance improvement when compared with the previous implementation.</li>
<li>The elimination of platform dependencies, which simplifies deployment and servicing. For example, <code>libcurl</code> is no longer a dependency on .NET Core for macOS and .NET Core for Linux.</li>
<li>Consistent behavior across all .NET platforms.</li>
</ul>
<p>If this change is undesirable, on Windows you can still use <a href="system.net.http.winhttphandler?view=net-6.0" class="no-loc" data-linktype="relative-path">WinHttpHandler</a> by referencing it's <a href="https://www.nuget.org/packages/System.Net.Http.WinHttpHandler/" data-linktype="external">NuGet package</a> and passing it to <a href="system.net.http.httpclient.-ctor?view=net-6.0#system-net-http-httpclient-ctor(system-net-http-httpmessagehandler)" data-linktype="relative-path"><code>HttpClient</code>'s constructor</a> manually.</p>
<h3 id="configure-behavior-using-run-time-configuration-options">Configure behavior using run-time configuration options</h3>
<p>Certain aspects of <a href="system.net.http.httpclient?view=net-6.0" class="no-loc" data-linktype="relative-path">HttpClient</a>'s behavior are customizable through <a href="/en-us/dotnet/core/run-time-config/networking" data-linktype="absolute-path">Run-time configuration options</a>. However, the behavior of these switches differs through .NET versions. For example, in .NET Core 2.1 - 3.1, you can configure whether <a href="system.net.http.socketshttphandler?view=net-6.0" class="no-loc" data-linktype="relative-path">SocketsHttpHandler</a> is used by default, but that option is no longer available starting in .NET 5.</p>
